# bithumb_mm — 신규상장 MM 봇 레짐 검증 스펙

> **목적**: 빗썸 KRW 신규상장 코인이 "봇 단독 유동성 공급 레짐"에 들어가는지 판별하고,
> 그 레짐에서 **거래비용을 넘는 예측 가능한 가격 행동**이 실재하는지 검증한다.
> 이 문서는 *무엇을 어떤 순서로 검증할지*만 정의한다. 매매·수집기 코드는 아직 만들지 않는다.
>
> **위치**: `src/research/bithumb_mm/` (backtest `src/research/{study_id}` 컨벤션 준용)
> **성격**: 리서치 — 매매하지 않는다. "이게 실제로 그런가?"를 통계로 확인한다.

## 1. 핵심 가설 (Hypothesis)

> 상장 초기 관심이 소멸한 뒤 시장은 **소수 MM의 기계적 정책이 도는 폐루프**가 된다.
> 정책이 기계적이면 가격 행동도 예측 가능하다.

유저 5개 가설 (원문):
1. 신규 상장 코인이 너무 많다 → **breadth. 미세 엣지를 시스템 전략으로 만드는 엔진.**
2. 움직이는 것처럼 보이려면 봇이 있어야 한다 → **가정 아님. 4번 필터로 코인별 검증 대상.**
3. MM은 불법 아님, 개인은 계속 털린다 → **구조적 참. 단 우리 우위 근거 = "레짐 예측가능성" 하나뿐.**
4. 호가창 기준 하루 이상 관찰 → 봇/개인 판별 → **기술적으로 확실히 가능. 쉬운 축.**
5. 봇 경향 파악 → 로직 생성 → **가장 약하고 미검증. 여기가 kill point.**

### 우리가 이기는 유일한 근거 (잊지 말 것)
우리가 진입하면 우리도 판의 참여자다. MM의 속도·정보 우위는 우리에게도 적용된다.
개인보다 나은 근거는 **딱 하나**: 레짐을 판별해 "봇 단독 구간"만 골라, 기계적 정책의 예측가능성을 이용한다.
이게 무너지면 우리도 그냥 털리는 개인이다.

### 해자 (moat)
"비밀 지식"이 아니라 **"고래에게 너무 작아서 매력 없는 니치"**.
얇은 깊이 → 큰 자본 진입 불가 → 정교한 자본이 알면서도 안 건드림.
→ 부트스트랩 1인에게 최적의 해자. 단, 규모를 키우면 해자가 사라지는 구조임(수용).

## 2. 데이터 현실 — 검증 순서를 지배하는 하드 팩트

| 데이터 | 과거 조회 | 백테스트 | 비고 |
|---|---|---|---|
| **캔들 (OHLCV)** | ✅ `to` 페이지네이션 | ✅ 가능 | 상장 시점까지 분봉 확보 가능 |
| **L2 호가창 깊이** | ❌ 현재 스냅샷만 | ❌ 라이브 수집만 | MM 탐지 정수가 여기 있음 |
| **틱 체결** | ❌ 최근분만 | ❌ 라이브 수집만 | cancel/체결 관찰 필요 |

**결론**: L2 없이는 봇 탐지 정밀화 불가. 하지만 **가격기반 엣지 존재증명은 캔들만으로 1차 가능.**
→ 검증은 반드시 **"싼 캔들 PoC 먼저 → 통과 시 비싼 L2 수집"** 순서.
호가 수집기는 오늘부터 안 쌓으면 미래 상장분을 통째로 놓침(시간 비대칭 = moat).

## 3. 검증 항목 (Phased)

### PHASE 0 — 데이터 접근 확인 (선행)
- [ ] 빗썸 API 2.0 키(Access/Secret) 확보 및 JWT 인증 REST 호출 성공
- [ ] 신규상장 코인 목록/상장 시각을 어디서 얻는지 확정 (마켓코드 폴링 vs 공지)
- [ ] 캔들 `to` 페이지네이션으로 특정 코인 상장 이후 전체 분봉 백필 가능한지 확인
- [ ] WebSocket(`wss://ws-api.bithumb.com/websocket/v1`) orderbook/trade 스트림 수신 확인

### PHASE 1 — 캔들 PoC: 엣지 존재증명 (KILL GATE) ★
> **여기서 죽으면 L2로도 못 살림. 프로젝트 접거나 재설계.**
- [ ] 과거 빗썸 신규상장 코인 N개 수집 (분봉)
- [ ] "상장 → 급등 → 감쇠 → 레인지" 패턴이 통계적으로 실재하는가 (거래량 감쇠 곡선의 knee 탐지)
- [ ] 감쇠 레짐 구간에서 **수익률의 음의 자기상관(평균회귀)이 유의**한가 (p-value/신뢰구간)
- [ ] **현실적 비용모델**(빗썸 수수료 + 얇은 판 스프레드 + 슬리피지)을 씌워도 살아남는가
- [ ] "봇 단독 레짐"과 "그냥 죽은 코인(유동성 고갈)"을 데이터로 구분할 수 있는가 ← 반드시 확인

### PHASE 2 — 라이브 L2 수집기 (PHASE 1 통과 시에만)
- [ ] 상장 감지 → 자동 수집 시작 파이프라인
- [ ] orderbook + trade 스트림을 코인별 타임스탬프로 적재 (파일/DB)
- [ ] 데이터 무결성: 갭/재연결/스냅샷+실시간 병합 검증

### PHASE 3 — 봇 지문 판별기 (L2 데이터 축적 후)
> 호가창의 봇 시그니처 — 아래 중 3개 이상 동시 점등 시 "사람 아님" 거의 확정
- [ ] **호가 재생성 지연**: 레벨 체결 후 수십 ms 내 동일 자리 복원
- [ ] **주문 사이즈 양자화**: 사이즈 분포 엔트로피 비정상 저하, 라운드넘버 심리 군집 부재
- [ ] **Cancel-to-trade 비율(OTR)**: 취소가 체결의 수십~수백 배 (미라지 깊이)
- [ ] **호가 갱신 시계 규칙성**: 고정 주기(100ms/500ms) quote 깜빡임
- [ ] **재고 skew 대칭성**: 양방향 호가가 재고 방향으로 동반 기울었다 회귀
- [ ] **레퍼런스 마켓 추종 지연**: 글로벌 가격에 고정 lag 반응 (있는 경우)

### PHASE 4 — 로직화 (엣지 확정 후)
- [ ] 판별된 봇 레짐에서의 진입/청산 로직 설계
- [ ] 레짐-이탈 탐지기(깊이 급소멸/변동성 스파이크/한쪽 호가 붕괴) — 진입 로직만큼 중요
- [ ] 러그/상폐 꼬리위험 대비 포지션 사이징·하드스탑
- [ ] 코인당 수용량(capacity) 상한 추정

## 5. 접어둔 것 (dead ends)

- **김치 프리미엄**: 프로 데스크가 이미 흡수, 개인 규모 수익원 아님. 빗썸 단독 상장은 글로벌 짝이 없어 개념 성립도 안 함.

---

## 6. 주의사항

1. **Look-ahead 금지 (전략 전환 시)**: 리서치 단계는 사실확인이라 미래 데이터 허용하나,
   PHASE 4 로직은 반드시 과거 데이터만 사용. 결과가 과도하게 좋으면 최우선 의심.
2. **판별 ≠ 수익**: 봇 판별(PHASE 3)은 필요조건이지 충분조건이 아님. 확신을 판별에 다 싣지 말 것.
3. **btc_backtest 읽기 전용**: 참조만, 절대 수정 금지.
